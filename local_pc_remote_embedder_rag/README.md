# Local PC RAG (Remote Embedder)

Standalone version that runs locally on your PC, while embeddings are always generated by your embedder server.

## What this contains

- Local API + UI: `FastAPI` (`web_server.py`, `run_server.py`)
- Training/indexing: `train.py` / `build_index.py`
- Retrieval + chat-like response: `retriever.py`
- Browser UI: `web/index.html`
- Windows scripts: `start_local.bat`, `train_local.bat`
- macOS/Linux scripts: `start_local.sh`, `train_local.sh`

## 1) Requirements

- Python `3.10` - `3.12` (recommended `3.11` on your machine)
- Running Qdrant (`local` or remote)
- Reachable embedding server endpoint

### Start Qdrant locally

Use Docker Compose from this folder:

```bash
docker compose up -d qdrant
```

If your machine uses the legacy command:

```bash
docker-compose up -d
```

### Start Qdrant with Colima (macOS)

```bash
chmod +x start_qdrant_colima.sh
./start_qdrant_colima.sh
```

Manual Colima flow:

```bash
colima start --cpu 4 --memory 6 --disk 40
docker context use colima
docker-compose up -d
```

Qdrant will be available at `http://127.0.0.1:6333`.

If you prefer a plain Docker run:

```bash
docker run -p 6333:6333 -p 6334:6334 -v qdrant_storage:/qdrant/storage qdrant/qdrant:latest
```

## 2) Configure environment

Copy and edit `.env.example`:

```bash
cp .env.example .env
```

Same setup as your main RAG server:
- If `../.env` exists (parent project), this local package automatically inherits it.
- If `./.env` exists (inside `local_pc_remote_embedder_rag`), it overrides parent values.
- Shell-exported variables still work as usual.

Required values in `.env`:

- `EMBEDDING_API_URL`
- `EMBEDDING_API_KEY`
- `EMBEDDING_API_MODEL`
- `IDIOM_DATA_ROOTS` (comma-separated folders/files with your CSVs)
- `QDRANT_URL` should point to your local Qdrant, usually `http://127.0.0.1:6333`

For your current setup, keep these values:

```dotenv
QDRANT_URL=http://127.0.0.1:6333
QDRANT_COLLECTION=JezikovneTehnologije
QDRANT_API_KEY=<your qdrant api key>
```

If local Qdrant has no API key enabled, leave:

```dotenv
QDRANT_API_KEY=
```

Use the same embedder URL as your main RAG server project.
If you already use these variables there, copy the same values here:

```bash
EMBEDDING_API_URL=...
EMBEDDING_API_KEY=...
EMBEDDING_API_MODEL=...
```

Bearer token notes:
- The app sends `Authorization: Bearer <token>` to your embedder.
- You can set any of these (first non-empty is used):
	- `EMBEDDING_API_KEY`
	- `EMBEDDING_BEARER_TOKEN`
	- `BEARER_TOKEN`

For your setup (local package + root CSVs with translations), use:

```bash
IDIOM_DATA_ROOTS=./data,..
```

This includes `english_idioms_optimized.csv` and any other CSV in the parent folder.

## 3) Train (build index)

### macOS/Linux

```bash
chmod +x train_local.sh
./train_local.sh
```

### Windows (cmd)

```bat
train_local.bat
```

Equivalent direct command:

```bash
python train.py --recreate-collection
```

If your embedder occasionally returns server-side `NaN` errors for some enriched texts,
training now automatically retries those rows with idiom-only text (no row is dropped).
The training output includes `fallback_embed_count` so you can see how many rows used fallback.

## GUI (3 things)

The UI has exactly these sections:

1. `Embedd (Train / Reindex)`
2. `Give an idiom to translate`
3. `Some tools / status` (health, current config/status)

For translation, it now retrieves using bilingual embedding text:
- English idiom
- meaning
- Slovene translation

This improves cases where the Slovene side is also idiomatic.

It also uses hybrid ranking:
- vector similarity from embeddings
- lexical overlap (query vs idiom/meaning/translation)
- exact/prefix/contains idiom boosts

You can tune it with:
- `RETRIEVAL_HYBRID_VECTOR_WEIGHT`
- `RETRIEVAL_HYBRID_LEXICAL_WEIGHT`
- `RETRIEVAL_EXACT_MATCH_BOOST`
- `RETRIEVAL_PREFIX_MATCH_BOOST`
- `RETRIEVAL_CONTAINS_MATCH_BOOST`

## 5) API endpoints

- `GET /api/health`
- `POST /api/train`
- `POST /api/chat`

`POST /api/chat` returns:
- `best_translation`: cleaned translation from top retrieved matches
- `postprocessed_translation`: best translation aligned to optional `llm_output`

Example request body for LLM post-processing:

```json
{
	"message": "A bad hair day",
	"llm_output": "To pomeni slab dan.",
	"history": []
}
```

## CSV format notes

Your CSV should include an idiom-like column (`idiom`, `expression`, `phrase`, etc.).
Optional columns: meaning (`meaning`, `definition`) and translation (`translation`, `target_translation`, `prevod`).
Columns like `slovene_translation` are automatically detected and used in training.
